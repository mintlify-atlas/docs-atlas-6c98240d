Nsengiyumva Eric is a full stack developer who has spent the last 4+ years specializing in backend engineering — building the APIs, services, and data layers that power production applications. He works across the full stack when needed, but his deepest expertise lives in server-side systems: the kind that stay up under load, respond fast, and fail gracefully when something goes wrong.

## Background

Eric's path into software started on the backend and has stayed there by choice. He reached for Node.js and TypeScript early, found NestJS as the framework that matched how he thinks about structure, and built up a pattern library of battle-tested approaches to authentication, data modeling, and service design.

His stack today reflects 4+ years of building for production:

- **Backend runtime:** Node.js, TypeScript — his primary environment for server-side work
- **Frameworks:** NestJS for structured, modular APIs; Express for lightweight services and middleware-heavy use cases
- **Authentication:** JWT and OAuth2 flows implemented across multiple production systems
- **Frontend:** React and Next.js for full-stack features, SSR, and internal tooling — not his primary focus, but a real part of his work
- **Databases:** PostgreSQL for relational data, MongoDB for document stores, MySQL for legacy and e-commerce contexts, Redis for caching and rate limiting
- **Currently learning:** Java and the JVM ecosystem — building toward cross-platform backend capabilities with Kotlin

Beyond shipping code, Eric contributes to open-source projects, mentors junior and mid-level developers, and writes technical guides through his [DevNotes blog](/blog/overview). He believes that the practices you share publicly become the practices that make the broader community stronger.

## Engineering philosophy

Eric's work is shaped by a set of practices he returns to across every project and every codebase.

**Clean, intentional APIs.** A well-designed API is a contract. Eric designs endpoints with consumers in mind — consistent naming, predictable error shapes, and documentation that keeps the integration experience from being a guessing game. The same thinking applies to internal module interfaces: clear inputs, clear outputs, no surprises.

**Modular design with NestJS primitives.** Eric uses NestJS guards, interceptors, and middleware not as boilerplate but as architectural tools. Authentication concerns belong in guards. Cross-cutting logic — logging, request transformation, error normalization — belongs in interceptors. Middleware handles the edges. Keeping concerns separated at this level means any part of the system can be tested, swapped, or scaled independently.

**Performance as a first-class requirement.** Eric approaches performance work the same way he approaches debugging: measure first, then act. That means query plan analysis before adding indexes, profiling before refactoring hot paths, and load testing before a service goes to production. Assumptions about performance are expensive; measurements are cheap.

**Production-grade reliability.** A service that deploys is not a service that's ready. Eric builds in health checks, structured logging, graceful shutdown, and circuit-breaking patterns from the start — not as a final phase but as part of the initial design. Systems that are observable from day one are systems that can be fixed at 2am without a crisis.
